## When to use this

You want to upload a presentation from somewhere other than the dashboard or a Claude session. For example:

- A CI/CD job that publishes a release deck on every tag
- A cron task that re-publishes a daily metrics deck
- Any custom integration

The interactive path is documented in [Use Slideless with Claude](/guides/with-claude). This guide is the script-friendly path.

## Prerequisites

- Node.js 20+ on the runner
- An **organization API key** with `presentations:write` (see [API keys](/concepts/api-keys))
- The HTML you want to share, ≤ 10 MB

## Install the CLI on the runner

```bash
npm install -g slideless
```

If global install is awkward in your CI environment, use `npx slideless ...` instead. It works identically.

## Authenticate non-interactively

In CI, pass the key through an environment variable. The CLI reads `SLIDELESS_API_KEY` automatically when no profile is configured:

```bash
export SLIDELESS_API_KEY="cko_your_key_here"
```

Or save it to a profile in non-interactive mode:

```bash
slideless login --api-key "$SLIDELESS_API_KEY" --name ci
```

## Publish

```bash
slideless share ./deck.html --title "Q4 review" --json
```

Output (`--json`):

```json
{
  "success": true,
  "data": {
    "shareId": "0192f1c3-...",
    "tokenId": "0192f1c3-...",
    "token": "AbCdEf...",
    "shareUrl": "https://app.slideless.ai/share/0192f1c3-...?token=AbCdEf..."
  }
}
```

Pipe through `jq` to extract the share URL:

```bash
SHARE_URL=$(slideless share ./deck.html --title "Q4 review" --json | jq -r '.data.shareUrl')
echo "Published: $SHARE_URL"
```

The exit code is non-zero on failure, so the CI step fails fast if the upload errors. Inspect `.error.code` for a stable machine-readable reason.

## Re-publish to the same URL

Save the `shareId` somewhere durable (a CI variable, a file in the repo, a secret store). Then:

```bash
slideless update "$SHARE_ID" ./deck.html --json
```

See [Update a presentation in place](/guides/updating-in-place) for the full pattern.

## Two URLs, two purposes

Don't confuse the **share URL** (what you give recipients) with the **fetch URL** (where the raw HTML lives):

| Purpose | URL | When |
|---|---|---|
| **Share URL** for recipients | `https://app.slideless.ai/share/{shareId}?token={token}` | What `slideless share` returns. Send this to people. |
| **Fetch URL** for raw HTML | `https://europe-west1-slideless-ai.cloudfunctions.net/getSharedPresentation/{shareId}?token={token}` | Used internally by the share viewer iframe. You rarely need this. |

If you embed the deck in an iframe yourself, use the fetch URL (it returns `text/html`). For everything else, use the share URL.

## Errors

The CLI maps backend HTTP statuses to short codes. Common ones for upload:

| `code` | Meaning |
|---|---|
| `unauthenticated` | Missing or invalid key |
| `permission-denied` | Key valid but missing `presentations:write` |
| `payload-too-large` | HTML exceeds 10 MB |
| `internal` | Backend 5xx; retry with backoff |

## Without the CLI

If you cannot install Node on the runner, hit the HTTP API directly. Reference: [`POST /uploadSharedPresentation`](/api-reference/upload-presentation).

```bash
curl -sS -X POST \
  -H "Authorization: Bearer $SLIDELESS_API_KEY" \
  -H "Content-Type: application/json" \
  --data-binary @<(jq -Rs --arg title "Q4 review" '{html: ., title: $title}' < ./deck.html) \
  https://europe-west1-slideless-ai.cloudfunctions.net/uploadSharedPresentation
```

The `jq -Rs` reads the file as a single JSON-escaped string. Don't try to inline raw HTML into a JSON payload, quotes and backslashes will break it.
