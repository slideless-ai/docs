## When to use this

You want to upload a presentation from somewhere other than the dashboard or a Claude session. For example:

- A CI/CD job that publishes a release deck on every tag
- A cron task that re-publishes a daily metrics deck
- Any custom integration

The interactive path is documented in [Use Slideless with Claude](/guides/with-claude). This guide is the script-friendly path.

## Prerequisites

- Node.js 20+ on the runner
- An **organization API key** with `presentations:write` (see [API keys](/concepts/api-keys))
- The deck you want to share — a folder or a single `.html` file, within your plan's size caps (see [Presentations](/concepts/presentations#size-and-file-count-caps))

## Install the CLI on the runner

```bash
npm install -g slideless
```

If global install is awkward in your CI environment, use `npx slideless ...` instead.

## Authenticate non-interactively

Pass the key through an environment variable. The CLI reads `SLIDELESS_API_KEY` automatically when no profile is configured:

```bash
export SLIDELESS_API_KEY="cko_your_key_here"
```

Or save it to a profile in non-interactive mode:

```bash
slideless login --api-key "$SLIDELESS_API_KEY" --name ci
```

## Publish a folder

```bash
slideless share ./my-deck --title "Q4 review" --json
```

Output:

```json
{
  "success": true,
  "data": {
    "shareId": "0192f1c3-...",
    "tokenId": "0192f1c3-...",
    "token": "AbCdEf...",
    "version": 1,
    "shareUrl": "https://app.slideless.ai/share/0192f1c3-...?token=AbCdEf...",
    "assetsUploaded": 5,
    "assetsDeduped": 0,
    "totalBytes": 194000
  }
}
```

Extract the share URL:

```bash
SHARE_URL=$(slideless share ./my-deck --title "Q4 review" --json | jq -r '.data.shareUrl')
echo "Published: $SHARE_URL"
```

The exit code is non-zero on failure, so the CI step fails fast if the upload errors.

### Single-file alternative

```bash
slideless share ./deck.html --title "Q4 review" --json
```

Both forms work interchangeably — same endpoints, same response shape.

### Strict mode for CI

In CI you probably want unresolved references (typos, missing files) to fail the build rather than just warn:

```bash
slideless share ./my-deck --title "Q4 review" --strict --json
```

## Re-publish to the same URL

Save the `shareId` somewhere durable (a CI variable, a file in the repo, a secret store). Then:

```bash
slideless update "$SHARE_ID" ./my-deck --json
```

Only changed files re-upload — the rest are deduplicated. See [Update a presentation in place](/guides/updating-in-place) for the full pattern.

## The upload protocol under the hood

`slideless share` and `slideless update` are thin wrappers around three HTTP endpoints:

1. **[`POST /precheckAssets`](/api-reference/precheck-assets)** — send a list of SHA-256 hashes, get back which ones the backend doesn't have.
2. **[`POST /uploadPresentationAsset`](/api-reference/upload-asset)** — multipart upload of one missing blob (content-addressed; server re-verifies the hash).
3. **[`POST /commitPresentationVersion`](/api-reference/commit-presentation-version)** — send the manifest (list of `{path, sha256, size, contentType}`), backend validates every hash exists, writes the manifest, bumps `currentVersion`.

The CLI runs them in order and prints progress between each step. See the linked API reference pages for the request/response shapes if you ever need to implement the flow in a language that isn't Node.

## Errors

The CLI maps backend HTTP statuses to short codes. Common ones:

| `code` | Meaning |
|---|---|
| `unauthenticated` | Missing or invalid key |
| `permission-denied` | Key valid but missing `presentations:write` |
| `payload-too-large` | File or total deck exceeds the plan cap |
| `too-many-files` | Manifest exceeds the plan's max file count |
| `blobs-missing` | Manifest references a hash not in the store (shouldn't happen in normal use — indicates the CLI or a custom client skipped a step) |
| `invalid-argument` | Bad path, bad sha256 format, etc. |
| `internal` | Backend 5xx; retry with backoff |

Static-scan errors emit `invalid-argument` with a details list of the offending references (e.g. `../outside/foo.jpg`).

## Two URLs, two purposes

Don't confuse the **share URL** (what you give recipients) with the **direct deck URL** (what the dashboard iframe embeds):

| Purpose | URL | When |
|---|---|---|
| **Share URL** for recipients | `https://app.slideless.ai/share/{shareId}?token={token}` | What `slideless share` returns. Send this to people. |
| **Direct deck URL** for iframing | `https://europe-west1-slideless-ai.cloudfunctions.net/getSharedPresentation/{shareId}/_t/{token}/` | Loads the entry HTML; relative asset paths (`./foo.jpg`) resolve natively. |

The share URL wraps the direct deck URL in chrome (download button, fullscreen toggle, footer). The direct URL is the raw rendering surface.
