title: "Update a presentation in place"
description: "Replace the content behind an existing share URL with one CLI command. Same URL, new content, view counts preserved, unchanged assets deduplicated."
---

## When to use this

You've already shared a deck. You want to change the content (fix a typo, update a number, add a slide, swap an image) without:

- Sending a new URL to everyone
- Losing the view counts and per-token analytics
- Re-uploading assets that haven't changed

This is what `slideless update` is for.

## What stays the same

| Stays the same | What changes |
|---|---|
| `shareId` | `currentVersion` (auto-bumps by 1; history is kept) |
| Every share token (same URLs) | `updatedAt` |
| `totalViews` and per-token `accessCount` | The deck contents |
| Owner, organization, title (unless you pass a new one) | |
| Tokens pinned to an earlier version (they stay pinned) | Tokens on `latest` auto-follow |

## Update in one command

```bash
# Folder
slideless update <shareId> ./my-deck

# Single file
slideless update <shareId> ./deck.html
```

Same URL. New content. The CLI prints the bumped version, the unchanged share URL, and how many assets were re-uploaded vs deduplicated.

```text
✓ Updated to version 4

  Share ID:        0192f1c3-...
  Version:         4
  Assets uploaded: 1
  Assets deduped:  12
  Total bytes:     194 MB
  Share URL:       https://app.slideless.ai/share/0192f1c3-...?token=...
```

Rename at the same time:

```bash
slideless update <shareId> ./my-deck --title "Q4 Pitch v2"
```

For machine-readable output:

```bash
slideless update <shareId> ./my-deck --json
```

```json
{
  "success": true,
  "data": {
    "shareId": "0192f1c3-...",
    "version": 4,
    "shareUrl": "https://app.slideless.ai/share/0192f1c3-...?token=...",
    "assetsUploaded": 1,
    "assetsDeduped": 12,
    "totalBytes": 194194304
  }
}
```

## Content-addressed dedup

Updates are cheap. The CLI hashes every file in the new deck, asks the backend which hashes it already has, and uploads only the missing ones. If you swap one image on a 200 MB deck, the network transfer is ~the size of that one image, not 200 MB.

```
# First upload (cold cache)
  Assets uploaded: 13
  Assets deduped:  0

# Second upload (same deck, nothing changed)
  Assets uploaded: 0
  Assets deduped:  13

# Third upload (one image swapped)
  Assets uploaded: 1
  Assets deduped:  12
```

## Pin before you update

If you shipped v3 to a specific person and want to make sure they keep seeing v3 no matter what you push next, **pin their token first**:

```bash
slideless pin <shareId> <tokenId> --to-version 3
```

Then update freely. The pinned recipient stays on v3. Unpin them later with `--latest` if you want them back on auto-update. See [Versioning](/concepts/versioning).

## Walkthrough

### 1. Find the shareId

If you saved the response from `slideless share`, the `shareId` is in there. Otherwise list your decks:

```bash
slideless list
```

Or pipe through `jq`:

```bash
slideless list --json | jq -r '.data.presentations[] | "\(.title) \(.id)"'
```

The shareId is also embedded in the share URL: `https://app.slideless.ai/share/{shareId}?token=...`.

### 2. Re-publish

```bash
slideless update 0192f1c3-... ./my-deck
```

### 3. Notify recipients (optional)

Recipients on `latest` tokens see the new content next reload. Pinned recipients stay on their version. The URL never changes.

## What can't be updated

- **Archived presentations.** Once archived, no updates accepted (`code: "archived"`).
- **Presentations you don't own.** The CLI returns `code: "permission-denied"`. The API key's user must match the owner.
- **A version that already exists.** Versions are append-only; you can't re-upload as v2 if v3 has already been committed. The next version is always `currentVersion + 1`.

## Errors

| `code` | Meaning |
|---|---|
| `unauthenticated` | Missing or invalid API key |
| `permission-denied` | Not the owner, or missing `presentations:write` scope |
| `not-found` | No presentation with that shareId |
| `archived` | Presentation has been archived |
| `payload-too-large` | File or total deck exceeds the plan cap |
| `blobs-missing` | Manifest references an unuploaded hash (CLI handles this automatically — only happens with custom clients) |

## Operational patterns

**CI on tag push.** A GitHub Action that re-publishes a release deck whenever you push a new tag. Same URL on the website, fresh content every release:

```bash
slideless update "$SLIDELESS_SHARE_ID" ./dist/release-deck --json --strict
```

**Daily cron.** A scheduled job that regenerates a metrics deck every morning and re-publishes it. If the graphs are the only thing changing, dedup means almost nothing re-uploads.

**Editorial workflow.** Author drafts, reviewer comments, author re-publishes. The shared URL given to stakeholders never changes. Pin the stakeholder's token before the reviewer round so mid-review fixes don't leak out.
