title: "Update a presentation in place"
description: "Replace the HTML behind an existing share URL with one CLI command. Same URL, new content, view counts preserved."
---

## When to use this

You've already shared a deck. You want to change the content (fix a typo, update a number, add a slide) without:

- Sending a new URL to everyone
- Losing the view counts and per-token analytics
- Breaking links you've already pasted in Slack, emails, or docs

This is what `slideless update` is for.

## What stays the same

| Stays the same | What changes |
|---|---|
| `shareId` | `version` (auto-bumps by 1) |
| Every share token (same URLs) | `updatedAt` |
| `totalViews` and per-token `accessCount` | The HTML content |
| Owner, organization, title (unless you pass a new one) | |

## Update in one command

```bash
slideless update <shareId> ./deck-v2.html
```

Same URL. New content. The CLI prints the bumped version and the unchanged share URL.

To rename at the same time:

```bash
slideless update <shareId> ./deck-v2.html --title "Q4 Pitch v2"
```

For machine-readable output:

```bash
slideless update <shareId> ./deck-v2.html --json
```

```json
{
  "success": true,
  "data": {
    "shareId": "0192f1c3-...",
    "version": 2,
    "shareUrl": "https://app.slideless.ai/share/0192f1c3-...?token=..."
  }
}
```

## Walkthrough

### 1. Find the shareId

If you saved the response from `slideless share`, the `shareId` is in there. Otherwise list your decks:

```bash
slideless list
```

Or pipe through `jq`:

```bash
slideless list --json | jq -r '.data.presentations[] | "\(.title) \(.id)"'
```

The shareId is also embedded in the share URL: `https://app.slideless.ai/share/{shareId}?token=...`.

### 2. Re-publish

```bash
slideless update 0192f1c3-... ./deck-v2.html
```

Output includes the bumped `version` and the same `shareUrl`.

### 3. Notify recipients (optional)

Recipients see the new content the next time they reload. If the deck is open in their browser already, they need to refresh. The URL itself doesn't change.

## What can't be updated

- **Archived presentations.** Once archived, no updates accepted (CLI returns `code: "archived"`).
- **Presentations you don't own.** The CLI returns `code: "permission-denied"`. The API key's user must match the owner.

## Errors

| `code` | Meaning |
|---|---|
| `unauthenticated` | Missing or invalid API key |
| `permission-denied` | Not the owner, or missing `presentations:write` scope |
| `not-found` | No presentation with that shareId |
| `archived` | Presentation has been archived |
| `payload-too-large` | HTML exceeds 10 MB |

## Operational patterns

**CI on tag push.** A GitHub Action that re-publishes a release deck whenever you push a new tag. Same URL on the website, fresh content every release:

```bash
slideless update "$SLIDELESS_SHARE_ID" ./dist/release-deck.html --json
```

**Daily cron.** A Cloud Scheduler job that regenerates a metrics deck every morning and re-publishes it.

**Editorial workflow.** Author drafts, reviewer comments, author re-publishes. The shared URL given to stakeholders never changes; the content under it improves.

## Without the CLI

If you can't run the CLI, hit the API directly. Full reference: [`POST /updateSharedPresentation`](/api-reference/update-presentation).

```bash
SHARE_ID="0192f1c3-..."

curl -sS -X POST \
  -H "Authorization: Bearer $SLIDELESS_API_KEY" \
  -H "Content-Type: application/json" \
  --data-binary @<(jq -Rs --arg id "$SHARE_ID" '{shareId: $id, html: .}' < ./deck-v2.html) \
  https://europe-west1-slideless-ai.cloudfunctions.net/updateSharedPresentation
```
